## 项目当前形态

这是一个电商自然语言问数 Agent 项目，用户用中文提问，系统自动完成业务绑定、上下文召回、SQL 生成、SQL 校验执行、结果返回和记忆沉淀。

后端主体能力已经比较完整：

- FastAPI 提供 `/api/query` SSE 流式问数接口。
- LangGraph 负责问数链路编排，当前图已经收敛成更高层的概念节点。
- MySQL 存储元数据、会话、会话消息和结构化业务数据。
- Qdrant 存储长期 SQL 经验记忆，用于相似成功问题召回。
- 业务绑定采用“候选抽取 + 元数据裁决”的方向，避免直接信任 LLM 的业务判断。
- SQL 执行已经收拢到 `RunSqlTool` 边界，并保留 SQL 校验、执行、异常处理和审计信息。
- 会话和长期记忆已经具备用户隔离、元数据版本隔离、匿名用户跳过长期记忆等边界。

当前已验证过的后端能力包括：

- 单轮问数评估通过。
- conversation memory 最小评估通过。
- 成功 SQL 经验可以写入向量库，并在后续相似问题中召回。
- 常规单测和 lint 已通过。

## 前端当前形态

前端是 React + Vite + Tailwind 的单页聊天式问数界面，核心文件在 `frontend/src/`。

现在已经具备的基础能力：

- 用户可以输入自然语言问题。
- 前端通过 `/api/query` 调用后端 SSE 接口。
- 能消费 `conversation`、`progress`、`result`、`usage`、`error` 等事件。
- 能展示用户消息、助手消息、执行步骤、结果表格和 token / cost 用量。
- 前端会生成并保存本地 `user_id`。
- 前端会把当前 `conversation_id` 传给后端，用于同一浏览器会话内的追问。

当前前端更像一个可用的技术 Demo，还没有完全变成适合面试演示的问数产品界面。

## 本次前端交付目标

本次前端只做轻量 ChatGPT 形态，不做复杂功能。

必须完成：

- 左侧会话列表。
- 中间 ChatGPT 风格消息流。
- 底部固定输入框。
- SSE 流式状态反馈。
- 轻量进度显示。
- 新建会话、切换会话、继续历史会话。
- 从后端加载完整历史会话。
- 查询结果以表格形式自然嵌入助手回答。

不需要完成：

- 图表。
- 导出。
- 收藏。
- 分享。
- BI 筛选器。
- 复杂调试面板。
- 前端展示 SQL / trace / memory hit / business binding。

## 前端还缺什么

### 1. 完整历史会话加载

这个必须做。

后端已经有会话和消息存储能力，仓储层也有按用户列出会话、读取会话消息的基础能力。前端目前只在内存里维护当前 `conversation_id`，刷新页面后不容易继续旧会话，也看不到历史会话。

前端需要能从后端加载当前用户的历史会话，并在左侧会话栏里展示。这样多轮问数才像一个真实产品，而不是一次性接口测试。

### 2. ChatGPT 风格消息体验

当前消息区能展示问答，但整体仍偏 Demo。需要让它更像用户熟悉的 ChatGPT 类界面：消息层次清楚、输入区稳定、滚动自然、助手回答过程不突兀。

后端当前不是 token 级文本流，而是阶段事件加最终结果。前端可以先做到“状态流式 + 最终回答出现”，不用为了逐字输出改后端。

### 3. 轻量进度显示

进度要保留，但不要展示过多内部节点。用户只需要看到类似：

- 理解问题
- 检索上下文
- 生成查询
- 执行查询
- 返回结果

这类阶段足够支撑演示，也不会把页面做成调试台。

### 4. 会话管理

前端需要支持新建会话、切换会话、继续历史会话。会话标题可以先用第一条用户问题或后端已有标题。

这块是多轮记忆展示的关键。没有会话管理，后端已经做好的 conversation store 很难被用户感知。

### 5. 结果展示保持简单

结果表格要继续保留，但不需要立刻做图表、导出、复杂 BI 控件。

前端只需要把 SQL 查询结果自然嵌入助手回答里，让用户能直接看到答案和表格。空结果、错误结果、成功结果要有清楚区分。

### 6. 调试信息放后端日志

前端不展示 SQL、trace、memory hit、business binding 等调试信息。面试讲解时这些可以通过后端日志、eval 报告或接口调试证明，不应该污染主页面。

主界面只展示用户能理解的信息：当前问题、回答、进度、结果和错误提示。

### 7. 错误状态更友好

后端会区分安全拦截、业务无法解析、SQL 校验失败、SQL 执行异常、超时等情况。前端不需要展示内部细节，但需要让用户知道这次为什么没有得到结果，以及是否可以换一种问法继续。

### 8. 视觉和文案产品化

当前前端能跑通主流程，但整体仍偏工程 Demo。需要围绕“电商问数助手”做视觉、文案和交互打磨，让它更像一个真实产品，而不是接口测试页面。

另外，前端文件里的中文内容在部分终端输出中存在编码显示异常，需要在实际浏览器和编辑器里确认最终展示是否正常。

## 不要扩大的范围

本次任务就是前端轻量产品化，不要扩大到新的后端架构改造。

不要做：

- 重做后端 Agent 图。
- 新增长期记忆类型。
- 引入复杂权限系统。
- 做图表、导出、收藏、分享等 BI 功能。
- 在前端展示 SQL、trace、memory 等调试细节。
- 重做主评估集。

前端只需要做到：好看、流畅、会话清楚、进度可见、历史可继续，就足够支撑当前项目目标。
