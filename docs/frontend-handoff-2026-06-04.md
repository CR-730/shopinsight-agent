# 前端任务交接文档

日期：2026-06-04

## 当前结论

后端问数链路已经完成阶段性收口，下一阶段主要工作在前端。前端不需要重新定义 Agent 架构，重点是把后端已经具备的会话、记忆、观测、结果展示能力，以更适合演示和面试讲解的方式呈现出来。

这份文档只说明项目现状和前端缺口，不限定具体实现方式。

## 项目当前形态

这是一个电商自然语言问数 Agent 项目，目标是让用户用中文提问，系统自动完成业务绑定、上下文召回、SQL 生成、SQL 校验执行、结果返回和记忆沉淀。

后端主体能力已经比较完整：

- FastAPI 提供 `/api/query` SSE 流式问数接口。
- LangGraph 负责问数链路编排，当前图已经收敛成更高层的概念节点。
- MySQL 存储元数据、会话、会话消息和结构化业务数据。
- Qdrant 存储长期 SQL 经验记忆，用于相似成功问题召回。
- 业务绑定采用“候选抽取 + 元数据裁决”的方向，避免直接信任 LLM 的业务判断。
- SQL 执行已经收拢到 `RunSqlTool` 边界，并保留 SQL 校验、执行、异常处理和审计信息。
- 会话和长期记忆已经具备用户隔离、元数据版本隔离、匿名用户跳过长期记忆等边界。

当前已验证过的后端能力包括：

- 单轮问数评估通过。
- conversation memory 最小评估通过。
- 成功 SQL 经验可以写入向量库，并在后续相似问题中召回。
- 常规单测和 lint 已通过。

## 前端当前形态

前端是一个 React + Vite + Tailwind 的单页聊天式问数界面，核心文件在 `frontend/src/`。

现在已经具备的基础能力：

- 用户可以输入自然语言问题。
- 前端通过 `/api/query` 调用后端 SSE 接口。
- 能消费 `conversation`、`progress`、`result`、`usage`、`error` 等事件。
- 能展示用户消息、助手消息、执行步骤、结果表格和 token / cost 用量。
- 前端会生成并保存本地 `user_id`。
- 前端会把当前 `conversation_id` 传给后端，用于同一轮浏览器会话内的追问。

当前前端更像一个可用的技术 Demo，还没有完全变成一个适合展示多轮记忆和智能问数能力的产品界面。

## 前端还缺什么

### 1. 会话能力还不够显性

后端已经有会话和用户隔离，但前端目前主要只在内存里维护当前 `conversation_id`。刷新页面后，用户不容易继续上一次会话，也看不到已有会话记录。

下一阶段前端需要让“当前正在和哪个会话对话”更清楚，方便演示多轮问数能力。

### 2. 记忆能力对用户不可见

后端已经有两类记忆：

- 会话消息历史，用于当前会话上下文。
- 长期 SQL 经验记忆，用于相似成功问题召回。

但前端目前看不出来某次回答是否使用了会话上下文，也看不出来是否召回了历史成功 SQL 经验。面试展示时，这会让“记忆能力”不够直观。

### 3. 调试和观测信息不足

后端现在已经能产生比较丰富的中间状态，例如会话 ID、业务绑定、阻断原因、SQL、用量、执行阶段等。前端目前主要展示进度、结果和 usage，对排查问题还不够友好。

前端需要补足最小可观测能力，让一次问数为什么成功、为什么失败、有没有被安全拦截、有没有使用记忆，都更容易定位。

### 4. 执行步骤展示和后端新链路不完全对齐

后端图已经从早期细碎节点收敛为更高层的链路，例如上下文构建、业务绑定、上下文压缩、SQL 生成、SQL 执行等。

前端的步骤展示仍偏旧流程，容易让演示者和维护者误解当前后端的真实链路。

### 5. 结果展示还比较基础

当前结果主要以表格展示。对于电商问数产品来说，结果展示还可以更贴近分析场景，例如更清楚的空结果状态、数值格式、摘要表达、图表入口、导出入口等。

这些属于前端体验优化，不需要改变后端核心链路。

### 6. 错误和阻断状态表达还不够细

后端会区分安全拦截、业务无法解析、SQL 校验失败、SQL 执行异常、超时等情况。前端目前对错误的呈现还比较泛化。

下一阶段需要让用户更容易理解“这次为什么没有返回结果”，也让开发者更容易根据错误类型定位问题。

### 7. 多轮追问的演示感还不强

后端已经支持多轮会话，但前端目前没有专门突出追问过程、上下文继承、会话连续性的展示。

如果这个项目用于面试，前端应该能让面试官直观看到：

- 第一轮问了什么。
- 第二轮如何基于上下文追问。
- 当前会话 ID 是否延续。
- 后端是否利用了历史上下文或长期 SQL 记忆。

### 8. 前端文案和视觉还需要产品化

当前前端已经能跑通主流程，但整体仍偏工程 Demo。后续可以围绕“电商问数工作台”方向做视觉、文案和交互打磨，让它更像一个真实业务工具，而不是接口测试页面。

另外，前端文件里的中文内容在部分终端输出中存在编码显示异常，需要在实际浏览器和编辑器里确认最终展示是否正常。

## 本阶段不建议扩大的范围

下一阶段重点是前端优化，不建议把任务扩大到新的后端架构改造。

暂时不需要：

- 重新设计后端 Agent 图。
- 新增长期记忆类型。
- 引入复杂权限系统。
- 扩大工具体系。
- 重做主评估集。

前端只需要把已有后端能力展示清楚、调试方便、演示顺畅，就已经足够支撑下一阶段项目目标。
